# Day 3 - Assignment Answers

## 1. What is a packet?

A packet is a small unit of data that is transmitted over a network. When you send information across the internet, whether it's loading a webpage, sending an email, or downloading a file, the data is not sent as one massive block. Instead, it is broken down into many smaller pieces called packets. Each packet contains the actual data payload along with important metadata: a source IP address (where it's coming from), a destination IP address (where it's going), a source port, a destination port, and sequence information. These packets travel independently across the network, possibly taking different routes, and are reassembled at the destination. For SOC analysts, examining individual packets is critical because malicious activity like malware communication, data exfiltration, or intrusion attempts can often be detected by looking at packet-level traffic.

## 2. Why are packets used instead of sending one large block of data?

Packets are used for several important reasons. First, if a large block of data were sent and even a single part was corrupted or lost during transmission, the entire block would need to be resent. With packets, only the damaged or missing packet needs to be retransmitted, making the process much more efficient. Second, packets allow data to be routed dynamically across different paths on the network, which improves reliability and performance. If one route is congested or down, packets can take alternate paths. Third, breaking data into packets allows multiple users and applications to share the network fairly, as packets from different sources can be interleaved. Fourth, it reduces the memory requirements on both the sender and receiver, since neither needs to buffer the entire file at once. This packet-switching approach is the foundation of how the entire internet works.

## 3. What is TCP?

TCP (Transmission Control Protocol) is a transport layer protocol that provides reliable, connection-oriented communication between devices on a network. Before sending any data, TCP establishes a connection using a three-way handshake (SYN → SYN-ACK → ACK) to ensure both the sender and receiver are ready. During transmission, TCP assigns sequence numbers to every packet so they can be reassembled in the correct order at the destination. It also uses acknowledgments (ACKs) to confirm that each packet was received, and automatically retransmits any packets that are lost or corrupted. This built-in error checking, delivery confirmation, and ordered reassembly make TCP the go-to protocol for applications where data integrity is critical such as web browsing (HTTP/HTTPS), email (SMTP, IMAP), file transfers (FTP, SFTP), and secure shell access (SSH). The trade-off is that TCP has higher overhead and is slower than alternatives like UDP because of all the additional control mechanisms.

## 4. What is UDP?

UDP (User Datagram Protocol) is a transport layer protocol that provides fast, connectionless communication. Unlike TCP, UDP does not establish a connection before sending data, does not guarantee that packets arrive at their destination, does not check for errors or retransmit lost packets, and does not ensure packets arrive in any particular order. This makes UDP significantly faster and lighter in terms of overhead compared to TCP. UDP simply takes the data, packages it into a datagram, and sends it on its way without any follow-up. This protocol is ideal for applications where speed is more important than perfect reliability such as online gaming (where a dropped frame is acceptable but lag is not), live video and audio streaming (where slight quality loss is preferable to buffering delays), DNS lookups (which need to be fast and are small enough to fit in a single packet), and VoIP (Voice over IP) calls. UDP's minimal overhead makes it the right choice whenever real-time performance matters more than guaranteed delivery.

## 5. Name three differences between TCP and UDP.

**First difference — Reliability:** TCP guarantees reliable delivery of all packets. If a packet is lost, TCP detects it and retransmits it automatically. UDP provides no such guarantee; if a packet is lost, it's simply gone with no recovery mechanism.

**Second difference — Connection model:** TCP is connection-oriented, meaning it must establish a connection through the three-way handshake (SYN → SYN-ACK → ACK) before any data can be sent, and it formally closes the connection when done. UDP is connectionless. It fires packets at the destination without any setup or teardown, making it faster to initiate communication.

**Third difference — Ordering and error checking:** TCP ensures packets arrive in the correct order by using sequence numbers, and performs thorough error checking. UDP delivers packets in whatever order they arrive (if they arrive at all) and performs only minimal error checking. This fundamental difference is why TCP is used for things like web browsing and file transfers where every byte matters, while UDP is used for streaming and gaming where occasional data loss is acceptable but speed is critical.

## 6. What is a port?

A port is a numerical identifier (ranging from 0 to 65535) that is used to distinguish between different services or applications running on the same device. While an IP address identifies the device itself (like a building's street address), a port identifies the specific service or application on that device (like the apartment number within the building). For example, a single server with IP address `192.168.1.10` might simultaneously run a web server on port 80, an email server on port 25, and an SSH server on port 22. The port number tells the operating system which application should receive the incoming data. Ports 0-1023 are "well-known ports" reserved for standard services, ports 1024-49151 are "registered ports" assigned to specific applications, and ports 49152-65535 are "dynamic" or "ephemeral ports" temporarily assigned to client applications when they initiate connections.

## 7. Why do we need port numbers?

We need port numbers because a single device can run multiple network services simultaneously, and the network stack needs a way to direct incoming data to the correct application. Without ports, when a server received data, the operating system would have no way of knowing whether that data was meant for the web server, the email server, the SSH daemon, or any other service running on that machine. Port numbers solve this problem by acting as endpoints for communication. When a client connects to a server, it specifies both the server's IP address (to reach the right machine) and a port number (to reach the right service on that machine). On the client side, the operating system also assigns a temporary ephemeral port so that the server knows where to send the response back to. This two-part addressing system (IP + port) is what makes multiplexing — multiple simultaneous network services on a single device — possible.

## 8. Which protocol does HTTPS use?

HTTPS uses **TCP** (Transmission Control Protocol) as its transport layer protocol. HTTPS is essentially HTTP (Hypertext Transfer Protocol) wrapped inside a TLS/SSL encryption layer, and this entire stack rides on top of TCP. TCP is chosen because web traffic requires reliable delivery — every part of a webpage (HTML, CSS, JavaScript, images) must arrive completely and in order for the page to render correctly. The TLS handshake that establishes the encrypted connection also depends on TCP's reliable, ordered delivery to negotiate encryption parameters before any application data is sent. So the full stack looks like: Application (HTTPS/HTTP) → TLS/SSL (encryption) → TCP (transport) → IP (network routing).

## 9. Which port does HTTPS use?

HTTPS uses **port 443** by default. When you visit a website with `https://` in the URL, your browser automatically connects to the server on port 443. This is the internationally recognized standard port for HTTPS traffic defined by IANA (Internet Assigned Numbers Authority). Port 443 was chosen to be close to port 80 (HTTP) for historical convenience, and it is one of the most commonly seen ports in SOC investigations because virtually all modern secure web traffic flows through it. In enterprise environments, seeing unexpected outbound connections to port 443 from unusual IP addresses or at unusual times can be a red flag, as attackers sometimes use HTTPS to disguise malicious communications since the encrypted traffic is harder to inspect.

## 10. Why is Port 3389 considered sensitive in cybersecurity?

Port 3389 is the default port for **RDP (Remote Desktop Protocol)**, which allows users to graphically control a remote Windows computer as if they were sitting in front of it. This port is considered highly sensitive for several critical reasons. First, RDP provides full interactive access to a system — an attacker who gains access via RDP can do everything a legitimate user can do: install malware, exfiltrate data, create new user accounts, and pivot to other systems on the network. Second, RDP has historically been one of the most heavily targeted services by attackers, with vulnerabilities like BlueKeep (CVE-2019-0708) allowing unauthenticated remote code execution. Third, brute-force attacks against RDP are extremely common — attackers constantly scan the internet for exposed RDP services and attempt to guess passwords. Fourth, many organizations inadvertently expose RDP to the internet, making it a prime entry point for ransomware attacks. In SOC work, seeing unexpected RDP connections (especially from external IP addresses or to critical servers) is an immediate alert that warrants investigation. For these reasons, security best practices recommend never exposing RDP directly to the internet, using strong authentication (including MFA), and monitoring all RDP traffic closely.
